# 示例发现（供 Agent 参考判断）

## 什么算 Bug vs 什么不算

### 是 Bug

| 发现 | 严重度 | 原因 |
|------|--------|------|
| 点击「保存」按钮无任何响应，console 有 TypeError | P0 | 功能不可用 |
| 用户列表页 console 出现 `Failed to fetch` | P1 | 数据加载失败 |
| 空表单提交后页面刷新但无错误提示 | P1 | 缺少校验反馈 |
| 导航到 `/settings` 返回 404 | P0 | 路由配置错误 |
| 输入 `<script>alert(1)</script>` 后页面弹出 alert | P0 | XSS 安全漏洞 |
| 快速双击提交创建了两条相同数据 | P1 | 缺少防重复提交 |
| 表格最后一列被截断，无横向滚动 | P2 | 布局问题 |
| 按钮文案显示为 `button.save.label` | P1 | i18n key 未翻译 |

### 不是 Bug

| 发现 | 原因 |
|------|------|
| Console 有 React DevTools 提示 | 开发工具信息，非错误 |
| Network 有 `/api/auth/session` 返回 401 | 未登录时的预期行为 |
| 页面加载 1.5s | 在可接受范围内 |
| Favicon 404 | 极低优先级，不报告 |
| `[HMR]` 相关 console 日志 | 开发环境热更新，非错误 |
| CSS `@layer` 警告 | 浏览器兼容性提示 |

## 高质量 Finding 示例

```markdown
#### [BUG-001] 候选人列表页分页后数据为空

- **页面**：/candidates
- **严重度**：P1
- **复现步骤**：
  1. 导航到 /candidates
  2. 确认第一页有数据显示
  3. 点击「下一页」按钮
  4. 观察页面内容
- **预期结果**：显示第二页候选人数据
- **实际结果**：表格显示「暂无数据」，但分页器显示共 3 页
- **Console**：无错误
- **Network**：GET /api/candidates?page=2 返回 200，body 为 `{"data": [], "total": 45}`
- **分析**：后端分页参数可能使用 0-based，前端传递了 1-based 的 page 值
```

## 关键判断原则

1. **以用户视角判断**：用户会注意到吗？会影响使用吗？
2. **区分环境因素**：开发环境的 HMR、DevTools 日志不是 bug
3. **预期 401/403**：未登录时的认证拦截是正常行为
4. **关联分析**：单个 console warning 可能不是 bug，但如果伴随功能异常则需关注
5. **复现确认**：偶现问题至少尝试复现 2 次再报告
