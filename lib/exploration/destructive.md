# Round 3: 破坏性测试

## 目标

发现权限绕过、竞态条件、状态不一致等深层缺陷。

## 前提

仅在 Round 1 + Round 2 完成后执行。生产环境跳过本轮。

## 测试类别

### 1. 权限绕过

#### 未认证访问
- 清除 cookie / token 后直接访问受保护路由
- 验证：应重定向到登录页，不应显示任何数据

```
browser_evaluate → document.cookie = ''
browser_navigate → /protected-route
browser_snapshot → 检查是否重定向
```

#### 越权访问
- 如果系统有多角色（admin/user）：
  - 以普通用户身份直接访问 admin 路由
  - 修改请求中的用户 ID 尝试访问他人数据

#### URL 篡改
- 修改 URL 中的 ID 参数：`/users/1` → `/users/2`
- 尝试访问不存在的 ID：`/users/99999`
- 负数 ID：`/users/-1`
- 非数字 ID：`/users/abc`

### 2. 快速重复操作

#### 表单双击提交
- 快速连续点击提交按钮 2-3 次
- 验证：不应创建重复数据

```
browser_click → submit 按钮
browser_click → submit 按钮（立即再次点击）
检查列表页是否出现重复条目
```

#### 并发导航
- 快速在多个路由间切换
- 验证：无白屏、无 JS 错误、数据不错乱

### 3. 陈旧状态

#### 后退按钮
- 完成创建操作 → 点击浏览器后退
- 验证：不应重新提交表单（无重复创建）

#### 过期数据
- 打开编辑页 → 不操作 → 等待 30 秒 → 提交
- 验证：应正常保存或提示冲突

#### 标签页竞争
- 在两个标签页打开同一条数据的编辑页
- 在标签页 A 修改并保存
- 在标签页 B 修改并保存
- 验证：应有冲突检测或 last-write-wins 明确策略

```
browser_tabs → new（打开新标签页）
browser_navigate → 同一编辑页
# 标签页 A 操作
browser_tabs → select(0)
browser_fill_form → 修改数据 A
browser_click → 保存
# 标签页 B 操作
browser_tabs → select(1)
browser_fill_form → 修改数据 B
browser_click → 保存
# 验证结果
```

### 4. 删除操作验证

- 删除关联数据：删除被引用的父记录
- 验证：应有关联检查提示，或级联删除正确执行
- 删除确认：验证删除前有确认对话框
- 删除后恢复：检查是否有撤销机制

### 5. 网络异常模拟

- 在请求进行中刷新页面
- 验证：不应导致数据不一致

## 安全边界

- **禁止**在生产环境执行本轮测试
- 所有写操作使用 `[QA-Agent]` 前缀数据
- 发现安全漏洞立即标记为 P0
- 不执行真正的攻击行为（如 DoS、数据批量删除）
