# URL 爬取发现（通用回退方案）

## 适用场景

- 框架未识别
- 框架已识别但路由扫描结果不足
- 作为补充发现手段，与源码扫描结果合并

## 爬取流程

### Step 1: 导航到首页

```
browser_navigate → base URL
browser_snapshot → 获取页面快照
```

### Step 2: 提取所有链接

从 snapshot 中提取所有可点击元素：
- `<a href="...">` 链接
- 导航菜单项
- 侧边栏链接
- 按钮（可能触发路由跳转）

### Step 3: 过滤链接

保留：
- 同域链接（与 base URL 同域）
- 内部路径（以 `/` 开头的相对路径）

排除：
- 外部链接（其他域名）
- 锚点链接（`#` 开头）
- 资源链接（`.js`、`.css`、`.png` 等）
- API 端点（`/api/` 前缀，除非是文档页）

### Step 4: 逐页深入（最多 2 层）

```
对每个发现的链接：
  1. browser_navigate → 链接 URL
  2. browser_snapshot → 获取快照
  3. 提取新链接 → 加入待访问队列
  4. 记录页面标题 + 主要交互元素
```

深度限制：从首页出发最多 2 层跳转，避免无限爬取。
页面数限制：最多发现 50 个独立路由。

### Step 5: 登录态处理

如果首页重定向到登录页：
1. 标记需要登录
2. 从 CLAUDE.md 提取测试账号信息
3. 完成登录后重新开始爬取
4. 同时记录登录页本身为一个路由

## 输出格式

爬取完成后，输出路由列表：

```
/               → 首页（P0）
/login          → 登录页（P0）
/dashboard      → 仪表板（P0）
/users          → 用户列表（P0）
/users/123      → 用户详情（P0，动态路由）
/settings       → 设置页（P1）
```

## 注意事项

- 爬取过程中记录所有 console error 和 network error
- 这些错误本身就是潜在 bug，直接进入 findings
- 对 SPA 应用，注意等待路由切换完成（`browser_wait_for`）
